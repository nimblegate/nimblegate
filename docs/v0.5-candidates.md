# V0.5 frame candidates

Frames to add or refine for V0.5, drawn from the V0 spec's design intent
(destructive prevention + compliance drift). Pre-dogfooding shortlist:
this list should be re-prioritised by audit-log evidence once V0 has
absorbed a real day of use.

Each candidate lists: trigger surface(s), severity, and a one-line rationale.

---

## Tier 1: Catastrophic prevention (highest leverage)

These catch operations that are functionally irreversible once they land.

### `fs-safety/rm-rf-protected-paths`
- **Triggers:** git-wrap
- **Severity:** BLOCK
- **What:** Block `rm -rf` (or `rm -fr`, `rm -Rf`) targeting any of: `/`, `$HOME`, `~`, `/etc`, `/usr`, `/var`, `/opt`, or the project root.
- **Why:** The single most common terminal-history regret. A canonical table at `.appframes/_canonical/protected-paths.toml` lets projects extend the list.

### `command-safety/curl-pipe-shell`
- **Triggers:** git-wrap
- **Severity:** BLOCK
- **What:** Detect `curl <url> | sh`, `curl <url> | bash`, `wget -O- <url> | sh`, and the equivalents with `eval`. Block unless `--force-yes` with reason.
- **Why:** Remote code execution by convention. Even trusted publishers get compromised; piping unverified bytes through sh is the canonical regret.

### `security/no-hardcoded-credentials`
- **Triggers:** pre-commit, cli
- **Severity:** BLOCK
- **What:** Regex-scan staged files for: AWS access key pattern (`AKIA[0-9A-Z]{16}`), GitHub PAT (`ghp_[A-Za-z0-9]{36}`), Stripe key (`sk_live_[A-Za-z0-9]+`), generic high-entropy strings near `password`/`secret`/`token`/`api_key` identifiers.
- **Why:** Once a credential is pushed, rotation is the only remedy. Pre-commit catches it cheap.

### `security/no-private-keys-in-repo`
- **Triggers:** pre-commit
- **Severity:** BLOCK
- **What:** Block staging of files matching `id_rsa`, `id_ed25519`, `*.pem` with `BEGIN PRIVATE KEY` content, `*.p12`, `*.pfx`, `*.key` with PEM content.
- **Why:** Same blast radius as Tier 1 above; file-name + content heuristic together yield low false positives.

### `git-safety/no-amend-pushed-commits`
- **Triggers:** git-wrap
- **Severity:** BLOCK
- **What:** Block `git commit --amend` when the current commit's hash exists on the remote-tracking branch (`@{upstream}`).
- **Why:** Amending a pushed commit followed by force-push silently drops co-authors' downstream commits. One of the highest-impact "I didn't realize" git mistakes on shared branches.

## Tier 2: Compliance drift (web app correctness)

The other half of the V0 spec: mechanically-preventable web hygiene.

### `app-correctness/html-meta-required`
- **Triggers:** pre-commit, cli
- **Severity:** WARN
- **What:** Every committed `.html` file must have: `<meta charset>`, `<meta name="viewport">`, `<html lang="...">`, non-empty `<title>`, and `<link rel="canonical">` (latter is project-toggleable).
- **Why:** SEO + accessibility floor; trivially detected; impossible to add later without revisiting every page.

### `security/csp-headers-required`
- **Triggers:** pre-commit, cli
- **Severity:** WARN
- **What:** Project containing `index.html` must also have a CSP-defining file: `_headers` (Cloudflare Pages / Netlify), `netlify.toml`, `vercel.json`, or a `<meta http-equiv="Content-Security-Policy">` tag in the HTML. Verify the CSP isn't trivially `default-src *`.
- **Why:** The single highest-impact site security header; routinely forgotten on greenfield static sites.

### `security/no-document-write-user-input`
- **Triggers:** pre-commit, cli
- **Severity:** BLOCK
- **What:** Extend the V0 innerHTML check pattern to also flag `document.write(<non-literal>)`, `insertAdjacentHTML`, `outerHTML = <non-literal>`, and React's `dangerouslySetInnerHTML={{__html: <non-literal>}}`.
- **Why:** Same XSS class as innerHTML: V0's frame is the prototype; V0.5 should cover the full family.

### `app-correctness/img-alt-required`
- **Triggers:** pre-commit, cli
- **Severity:** WARN
- **What:** Every `<img>` in committed HTML must have a non-empty `alt` attribute (or explicit `alt=""` for decorative images, with project-toggleable behaviour).
- **Why:** A11y floor; mechanically detectable; cumulative drift if not enforced per-commit.

### `convention/robots-sitemap-link`
- **Triggers:** cli
- **Severity:** INFO
- **What:** If `robots.txt` exists, it should contain a `Sitemap:` line; if `sitemap.xml` exists, it should be referenced from `robots.txt`.
- **Why:** Indexing hygiene; trivial to check; never a false positive.

## Tier 3: Convention frames (team-share value)

Useful when more than one human (or one human + multiple AI agents) commit to the repo.

### `convention/commit-message-format`
- **Triggers:** pre-commit (commit-msg hook actually, but pre-commit is V0's only git hook)
- **Severity:** WARN
- **What:** Verify commit messages match Conventional Commits (`feat:`, `fix:`, `chore:`, `docs:`, `test:`, `refactor:`, `style:`, `perf:`, `build:`, `ci:`, optional scope, then summary).
- **Why:** Enables generated changelogs, semantic versioning, scoped filtering of `git log`.

### `convention/branch-name-format`
- **Triggers:** git-wrap
- **Severity:** WARN
- **What:** Verify branch names match a project-configurable regex (default: `^(main|master|develop|feat|fix|chore|docs)/.+`).
- **Why:** Sortable branch lists, automated CI routing by prefix.

### `convention/no-trailing-whitespace`
- **Triggers:** pre-commit
- **Severity:** INFO
- **What:** Staged text files must not contain trailing whitespace on any line.
- **Why:** Reduces noise in diffs; low-cost to enforce; not project-specific.

## Tier 4: Refinements to V0 frames

Not new frames; tightenings of existing ones based on the obvious gaps in V0.

- **`security/no-innerHTML-user-input`**: extend pattern to also catch `outerHTML`, `insertAdjacentHTML`, `document.write` (or split out per Tier 2's `no-document-write-user-input`).
- **`git-safety/folder-branch-lock`**: support per-subdir mappings (nested), not just first-level folders.
- **`git-safety/no-force-push-main`**: move `protectedBranches` from hardcoded var into `.appframes/_canonical/protected-branches.toml` so projects can extend it.
- **`command-safety/apt-purge-preview`**: generalise to `npm uninstall`, `yarn remove`, `pip uninstall`, `cargo remove`, any package manager's removal verb without a corresponding dry-run/preview verb.

## Tier 6: Documentation enforcement

Mechanically-preventable documentation drift. All convention-category, all
trigger off pre-commit + cli so they catch drift before it lands.

### `convention/doc-touches-with-code` (highest leverage)
- **Triggers:** pre-commit, cli
- **Severity:** WARN
- **What:** A canonical table `.appframes/_canonical/code-doc-map.toml` declares
  source-path → docs-path coupling. Pre-commit fails (WARN, not BLOCK) when a
  staged commit touches a source file but not the corresponding doc.
- **Why:** The single most reliable "is the doc current?" signal: if the doc
  moved with the code in the same commit, it's at least likely-current. False
  positives are owner-driven (the project author defines the map), so projects
  can dial precision.
- **Canonical table shape:**
  ```toml
  [code-to-docs]
  "internal/checks/*.go"   = "docs/frame-authoring.md"
  "cmd/nimblegate/main.go"  = "README.md"
  "internal/canonical/*.go" = "docs/canonical-tables.md"
  ```
- **Override:** commit message containing `[no-doc-update]` skips the check
  for that commit (recorded in audit log).
- **Sketch:** [`docs/sketches/frame-sketches/doc-touches-with-code.md`](sketches/frame-sketches/doc-touches-with-code.md).

### `convention/changelog-entry-per-commit`
- **Triggers:** pre-commit
- **Severity:** WARN
- **What:** Any commit touching paths under a configurable include-glob list
  (default: `cmd/**`, `internal/**`) must also stage at least one **added** line
  in `CHANGELOG.md`. Trailing/whitespace-only edits don't count.
- **Why:** Stops "what changed?" from accumulating in `git log` and becoming
  read-only. Cheap to author per commit; high payoff at release time.
- **Override:** commit message contains `[no-changelog]` (e.g. typo fixes,
  test-only commits, doc-only commits).
- **Acceptance:** test fixtures for: staged code + CHANGELOG → PASS; staged
  code, no CHANGELOG → WARN; staged code + `[no-changelog]` in message → PASS.

### `convention/dated-todo`
- **Triggers:** pre-commit, cli
- **Severity:** WARN
- **What:** Staged files scanned for `TODO`, `FIXME`, `HACK`, or `XXX`
  markers that lack at least one of:
  - ISO date with optional condition: `TODO(2026-06-15)`, `TODO(2026-06-15: ship V0.5)`
  - Owner: `TODO(@user)`, `FIXME(you)`
  - Issue link: `TODO(#42)`, `FIXME(SNAP-123)`
- **Why:** Naked TODOs accumulate forever; tagged ones are at least
  searchable + retire-able. The project's existing pre-commit catches them
  at write time, not after a year of build-up.
- **Override:** add expected tag, or use `// appframes:disable convention/dated-todo`.
- **Sketch:** [`docs/sketches/frame-sketches/dated-todo.md`](sketches/frame-sketches/dated-todo.md).

### `convention/public-symbol-doc-comment`
- **Triggers:** pre-commit
- **Severity:** INFO (start); promote to WARN after a clean dogfooding run.
- **What:** Language-specific: exported Go identifiers (capitalised funcs,
  types, vars, consts) must have a leading doc comment. Plug-in pattern for
  other languages (JSDoc, Python docstrings).
- **Why:** This is what `revive` / `golint` already does; embedding it as
  a frame means one less tool to install and the policy lives next to other
  project rules.
- **Note:** stdlib already has tools for this language-by-language. Frame
  version is useful for stack-mixed repos that want one gate.

### `convention/markdown-link-check-internal`
- **Triggers:** pre-commit, cli
- **Severity:** WARN
- **What:** Markdown files (`*.md`): verify every relative link `[text](./path)`
  or `[text](../path)` resolves to an existing file. External `http(s)://`
  links are explicitly NOT checked (too slow, too flaky).
- **Why:** Internal link rot is the dominant doc-quality bug; cheap to verify
  at commit time. External link check belongs in a scheduled job, not
  pre-commit.
- **Override:** `<!-- appframes:disable convention/markdown-link-check-internal -->`
  at top of file.

## Tier 5: Engine + tooling improvements (non-frame)

Code-level changes that aren't new frames. From the V0 test-suite-findings doc and ongoing observations.

### Canonical-table caching with incremental invalidation
- **Where:** `internal/canonical/tables.go` + every check that reads a canonical table.
- **Problem:** `FolderBranchLock` and `CrossBranchIDConsistency` re-read + re-parse `.appframes/_canonical/*.toml` on every invocation. Benchmark shows ~20µs per `FolderBranchLock` call dominated by disk read: fine at 1 check, wasteful at 100.
- **Design:**
  1. Add a per-engine `canonical.Cache` struct keyed by absolute table path.
  2. Each lookup `Cache.Load(path)` checks the file's `mtime` (cheap `Stat`).
     - If first-seen or `mtime` newer than cached entry → re-parse, replace entry.
     - Otherwise return the cached `Table`.
  3. Only the changed files re-load, incremental.
  4. Cache is engine-scoped (one per `engine.New`), so a long-lived process picks up edits within the same session.
- **Cache control surface:**
  - `nimblegate cache clear` subcommand: drops all in-memory cache + any future on-disk cache files. Useful when the user suspects stale data or a bug.
  - `nimblegate cache status`: shows what's cached, when each entry was last refreshed.
  - Audit log records cache invalidations as INFO entries (so `nimblegate watch` shows them).
- **Acceptance:** bench shows `FolderBranchLock` second-call latency drops from ~20µs to <2µs; first-call latency unchanged.
- **Out of scope for now:** on-disk persistent cache (would need its own integrity checks); cross-process cache sharing.

### `--force-with-lease` severity downgrade
- **Where:** `internal/checks/noforcepushmain.go`.
- **Problem:** `git push --force-with-lease` is the safer variant (atomic compare-and-set against remote). V0 treats it identically to `--force` and BLOCKs both. Users who know what they're doing end up using `--force-yes` more often than needed, polluting the audit log with overrides.
- **Design:**
  1. Detect `--force-with-lease` as a distinct flag.
  2. Outcome WARN (not BLOCK) when pushing `--force-with-lease` to a protected branch.
  3. Still BLOCK on bare `--force` / `-f`.
- **Acceptance:** new test cases in `internal/checks/noforcepushmain_test.go` verify both behaviours; docs/manual-tests/gitwrap.md updated.

### Canonical table validation: reject nested subtables
- **Where:** `internal/canonical/tables.go`.
- **Problem:** `[branches.protected]` style nested TOML produces values like `"map[no-force:true]"` when stringified: useless to check functions, confusing in errors. Test `TestLoad_NestedSubtableBehavior` documents the current weirdness.
- **Design (pick one):**
  - **Option A (strict):** at load time, if any top-level section contains a non-string value, return a validation error naming the section and offending key. Forces canonical tables to stay flat: one `[section]` deep, all values strings.
  - **Option B (expose nesting):** change `Table` to support nested maps and update consumers. More work but matches TOML semantics.
- **Recommendation:** Option A for V0.5: keeps canonical tables simple and prevents silent misconfiguration. Option B can wait for a real use case.
- **Acceptance:** `TestLoad_NestedSubtableBehavior` updated to assert the validation error; lint output gains a clear message for the project author.

## Decision criteria (apply after dogfooding)

For each candidate above, the V0.5 ship decision should require:
- At least one true-positive observed during V0 dogfooding (real catch in the audit log), OR
- A documented near-miss / incident in the dogfooding log that the frame would have prevented.

Frames in this list with neither qualify for V0.6+, not because they're wrong, but because there's no evidence they earn their cost in false positives.

## Frames to deliberately not build

Recorded so a future "why not add X?" question has an answer:

- **License-header-required**: too project-specific; templates vary widely; better as a project-local frame than stdlib.
- **Spell-check on commit messages / docs**: false-positive rate too high; better handled by editor.
- **Test-coverage threshold**: requires language-specific tooling already; out of scope for a stack-agnostic gate.
- **Style/lint rules** (gofmt, prettier, eslint, ruff): already solved by per-stack tools; nimblegate is the wrong layer.
